1. CORS 정책으로 인한 요청 차단

문제 개요

JWT 인증 API 호출 시 Authorization 헤더가 포함된 POST 요청이 CORS 정책에 인해 차단되었습니다.
로컬에서는 allowedOrigins로 허용했으나, 배포 환경에서는 브라우저가 요청을 거부했습니다.

해결 방법

@CrossOrigin은 컨트롤러 단위로만 적용되어 개별 설정 시 누락이 발생했습니다.
전역 CORS 설정(WebMvcConfigurer)을 적용하여 모든 경로에 동일한 정책이 반영되도록 수정했습니다.

코드

@Configuration
public class CorsConfig implements WebMvcConfigurer {
    @Override
    public void addCorsMappings(CorsRegistry registry) {
        registry.addMapping("/**")
            .allowedOrigins(
                "http://localhost:3000",
                "http://175.45.203.68:3000"
            )
            .allowedMethods("GET", "POST", "PUT", "DELETE")
            .allowedHeaders("*")
            .allowCredentials(true);
    }
}

교훈

@CrossOrigin은 컨트롤러 단위로 테스트나 제한적 API에 적합하며, 운영환경에서는 전역 정책 WebMvcConfigurer 방식이 유지보수와 보안 측면에서에서 더 안정적임을 배웟습니다.
실제 프로젝트 운영에는 WebMvcConfigurer 혹은 두가지 방식을 혼용하여 사용함을 배웠습니다.

2. Redis TTL 관리 개선

문제 개요

인증 코드 TTL을 관리하기 위해 Redis Hash 구조를 사용했으나, 필드 단위 TTL(Time To Live) 설정이 불가능했습니다.
일부 인증 코드가 만료 후에도 Redis에 남아있었고, TPS 부하 시 인증 실패율이 증가하며 응답 지연이 2.4초 이상 발생했습니다.

해결 방법

Hash 구조를 Key-Value(String) 구조로 전환하여 각 코드별 TTL을 개별 관리하도록 변경했습니다.

코드

redisTemplate.opsForValue().set(
  "email_auth:signup:" + email,
  code,
  1, TimeUnit.HOURS
);

교훈

TTL 제어가 중요한 데이터는 Hash 구조보다 String 구조가 적합하며,
대량 트래픽 환경에서는 TTL 관리 실패가 시스템 안정성에 직접적인 영향을 미침을 배웠습니다

3. 타임존 적용 오류

문제 개요

사용자 입력 시간이 DB에 저장될 때 9시간 느리게 저장되는 문제가 발생했습니다.
로컬 환경에서는 KST 기준으로 정상 동작하였으나, 배포 환경에서는 UTC 기준으로 저장되었습니다.
이는 Docker 컨테이너의 기본 시간대가 UTC로 설정되어 있었기 때문입니다.

해결 방법

JVM 실행 시 KST로 시간대를 지정하고, Dockerfile 내에서도 컨테이너 OS의 타임존을 KST로 설정했습니다.
이를 통해 DB와 로그 모두 동일한 KST 기준으로 정상 동작함을 확인했습니다.

코드

# JVM 실행 시 KST 설정
-Duser.timezone=Asia/Seoul

# Dockerfile 내 시간대 설정
java -Duser.timezone=Asia/Seoul -jar your-app.jar

교훈

서버 환경과 로컬 환경의 타임존 불일치는 시간 기반 기능에서 치명적인 오류를 유발할 수 있습니다.
Docker 기반 Java 애플리케이션 배포 시, JVM 시간대와 컨테이너 OS 시간대를 모두 명시적으로 설정해야 안정적으로 동작함을 배웠습니다.
또한, Jenkins 자동화 배포 환경에서도 두 설정이 충돌 없이 병행될 수 있음을 확인했습니다.
